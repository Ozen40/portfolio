Tests unitaires et fonctionnels

Ma définition :

Les tests unitaires et fonctionnels sont des pratiques qui permettent d'assurer la qualité et la fiabilité du code.
Les tests unitaires se concentrent sur la vérification du bon fonctionnement de petites parties du code (fonctions, méthodes, classes) de manière isolée, en utilisant des mocks pour simuler des dépendances.
Les tests fonctionnels permettent, quant à eux, de tester de manière plus globale le bon fonctionnement du code avec différents cas de test.


Mes éléments de preuve :

Lors de mon alternance dans le projet Grand Angle, j'ai appris à faire des tests unitaires sur Java à l'aide de JUnit et Mockito. J'ai également appris à structurer mes tests en trois parties grâce au design pattern “Arrange-Act-Assert”.
La partie “Arrange” est celle permettant de préparer les données qui vont être utilisées lors du test.
La partie “Act” quant à elle va s'occuper d'appeler la méthode que l'on veut tester.
Pour finir, la partie “Assert” permet de comparer les valeurs réelles avec les valeurs attendues.
De plus, lorsque du nouveau code est ajouté, il faut que ce dernier ait une couverture de test de minimum 60% ce qui signifie qu'au moins 60% du nouveau code doit être testé avec des tests unitaires mais je fais en sorte d'atteindre les 100% lorsque cela est possible.

Lors de mon stage dans le projet OpenENT, j'ai eu l'occasion de réaliser des tests fonctionnels sur plusieurs environnements. Par exemple, il m'est arrivé de tester une fonctionnalité liée aux dates et heures, et j'ai observé des résultats différents selon les bases de données utilisées. L'une d'elles était configurée en UTC (Universal Time Coordinated), tandis que l'autre utilisait l'heure de Paris, ce qui conduisait à des différences entre les résultats.


Mon autocritique :

Je me considère comme autonome dans la réalisation de tests unitaires et fonctionnels. Je suis capable de concevoir des tests qui valident le comportement attendu du code et je comprends l'importance de cette pratique afin d'obtenir un code de la meilleure qualité possible. J'ai cependant encore à apprendre sur les tests fonctionnels par exemple.

Selon moi, savoir réaliser des tests unitaires est une compétence à ne pas négliger lors de son apprentissage, car elle permet d'avoir rapidement un retour sur la fiabilité du code, d'identifier les régressions plus facilement et de faciliter la maintenance du projet sur le long terme.


Mon évolution dans cette compétence :

J'ai pour objectif d'atteindre un niveau confirmé dans cette compétence et pour ce faire, je compte apprendre à effectuer du Test Driven Development (TDD) qui est une approche de développement qui consiste à écrire les tests avant d'écrire le code réel. Je compte apprendre cette nouvelle approche avec le livre “Test Driven Development: By Example”.
