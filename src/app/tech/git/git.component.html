Git

Ma définition :

Git est un système de contrôle de version distribué, créé par Linus Torvalds en 2005, qui permet de suivre les modifications apportées au code source d'un projet informatique. Il offre aux développeurs la possibilité de travailler simultanément sur un même projet, de gérer différentes versions du code, de revenir à des états antérieurs si nécessaire et de fusionner les contributions de chacun. Contrairement aux systèmes de contrôle de version centralisés, Git permet à chaque développeur d'avoir une copie complète du dépôt avec tout l'historique des modifications sur sa machine locale. Selon le rapport 2022 de Stack Overflow, Git est utilisé par plus de 95% des développeurs professionnels, ce qui en fait l'outil de gestion de version standard de l'industrie.


Mes éléments de preuves :

Lors de mon stage dans le projet OpenENT (lien vers la réalisation), j'ai pu découvrir le travail collaboratif en entreprise mais également le fonctionnement d'un projet open source. J'ai appris à respecter les conventions de nommage des commits selon les normes du projet, à créer des branches thématiques pour chaque fonctionnalité, et à soumettre des Pull Requests (PR) qui étaient ensuite revues par les autres développeurs avant d'être mergé sur la branche principale.



J'ai continué mon apprentissage de Git lors de mon alternance dans le projet Grand Angle (lien vers la réalisation) dans lequel j'ai beaucoup appris sur la notion de versionning. En effet, lorsqu'un correctif était apporté, il était obligatoirement mergé sur la branche “develop” de l'application qui représente la dernière version disponible pour les clients (ex: 8.8). Cependant, en fonction de la criticité de l'anomalie, certains correctifs étaient “redescendus” dans des versions inférieures (ex: 8.7.X et 8.7.1.X) en faisant des cherry-picks des commits contenant les corrections.


Mon autocritique :

Je me considère comme ayant un niveau confirmé dans l'utilisation de Git. Je maîtrise les commandes essentielles comme commit, push et pull mais également certaines plus avancées comme merge, rebase ou cherry-pick. J'ai cependant encore un peu de mal à gérer les conflits de merge lorsqu'ils apparaissent. Je conseille à tous les développeurs d'apprendre Git, non seulement car c'est une compétence obligatoire en entreprise mais qui est également très utile pour des projets personnels ou si l'on veut participer à des projets open source.


Mon évolution dans cette compétence :

J'ai pour objectif d'atteindre un niveau expert sur Git et pour cela, j'ai prévu d'apprendre à créer mon propre dépôt Git mais également à créer des pipelines et d'utiliser des outils de CI/CD comme Jenkins. Je prévois donc de suivre le guide de Gitlab “Use CI/CD to build your application” qui explique comment fonctionnent les pipelines, les jobs mais également comment faire passer des tests à son code pour qu'il soit accepté.
